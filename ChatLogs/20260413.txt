› utf-8로 되어있거든? 그럼 이거 anis euc_kr로 해줘? 아님 어떤 형식으로 수정할까?


• 파일 자체는 UTF-8로 유지하는 쪽이 맞습니다. 지금 보인 깨짐은 문서 포맷 문제라기보다 제가 읽은 터미널/세션 쪽 디코딩 문
  제일 가능성이 큽니다.

  정리하면:

  - UTF-8 유지 권장
  - 가능하면 UTF-8 without BOM 권장
  - ANSI로 바꾸는 건 비추천
  - EUC-KR도 비추천
  - Windows에서 말하는 ANSI는 보통 한국어 환경에선 CP949라서, EUC-KR과도 완전히 같지 않습니다

  왜 UTF-8이 맞냐면:

  - Git, Markdown, VS Code, GitHub, 협업 툴 호환성이 가장 좋음
  - 나중에 다른 환경에서 다시 인코딩 문제 생길 가능성이 적음
  - Unity 프로젝트 문서도 보통 UTF-8로 관리하는 게 안전함

  예외적으로 ANSI/CP949를 고려할 상황:

  - 아주 오래된 메모장/툴만 써야 하고
  - GitHub/VS Code보다 로컬 Windows 전용 툴 호환이 더 중요할 때

  실무적으로는 이렇게 가면 됩니다:

  - a.md, DEVLOG.md, README.md는 UTF-8 유지
  - 필요하면 제가 PowerShell에서 UTF-8로 제대로 읽히게 환경만 맞춰서 다시 확인
  - 원하면 문서 저장 형식을 UTF-8 with BOM으로 통일하는 것도 가능
      - 아주 일부 Windows 환경에선 BOM 있는 쪽이 덜 깨집니다
      - 대신 일반적으로는 without BOM이 더 깔끔합니다

  제 추천:

  - 문서 인코딩은 UTF-8 그대로
  - 변환하지 말고
  - 제가 읽는 쪽을 UTF-8 기준으로 다시 맞춰서 확인

  1. UTF-8 기준으로 다시 읽어서 정상 표시 확인
  2. 세 파일을 UTF-8 with BOM으로 통일
